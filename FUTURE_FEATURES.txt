1.  PWA CONVERSION
=============================================================

WHAT
  Convert the FastAPI/Jinja webapp to a Progressive Web App so
  inspectors can install it to their phone home-screen and use it
  offline-first. The mobile-app endpoint already POSTs photos to
  /api/upload — PWA gives a unified surface for both office desktop
  and field-mobile use.

WHY
  - One codebase serves desktop + mobile field use
  - Offline-first: inspector can mark photos in poor signal areas,
    sync queue uploads when connectivity returns
  - Push notifications for "batch finished classifying"
  - Apple/Android home-screen install without app-store overhead

IMPLEMENTATION SKETCH
  1. Add /static/manifest.json with name, short_name, icons (192,
     512), theme_color (#10b981 — same as existing emerald theme),
     display: "standalone", start_url: "/".
  2. Add /static/service-worker.js. Two cache strategies:
       - precache for app shell (HTML + Tailwind CDN + app.css + JS)
       - runtime cache for /static/* + R2 photo thumbnails (LRU)
  3. <link rel="manifest" href="/static/manifest.json"> in _base.html
  4. <script>navigator.serviceWorker.register("/static/service-worker.js")</script>
  5. Apple-touch-icon meta + apple-mobile-web-app-capable for iOS
  6. Background sync API for queued uploads:
       - When /api/upload fails (offline), store the FormData in
         IndexedDB with a UUID
         - Service worker registers a 'sync' event
         - On reconnect, replay queued uploads
  7. Test install flow on Chrome (desktop), Safari (iOS), Chrome
     (Android). iOS has the most restrictive PWA story — single-page
     navigation only, limited storage.

DEPENDENCIES / CAVEATS
  - The existing dropzone + upload flow is already tolerant of
    chunked uploads; sync queue plumbs into it without restructuring.
  - HEIC handling on mobile: iOS share-sheet hands files as HEIC.
    The /api/upload endpoint already converts via pillow-heif. PWA
    install doesn't change that.
  - Tailwind CDN is a runtime dependency; precache it or self-host.

EFFORT
  ~2-3 days for a polished PWA. ~1 day for a basic install + cache.

2.  OAUTH + LOGIN (Google primary, Azure AD as alternative)
=============================================================

WHAT
  Replace the current "no auth, single tenant" model with proper
  OAuth. Implemented LAST in the feature sequence — every other
  feature below assumes auth exists.

  Provider chosen: Google OAuth (project gen-lang-client-0891682097,
  client "violation-bot"). Credentials staged at:
    ~/Desktop/violation-bot/secrets/client_secret_*.googleusercontent.com.json
  Azure AD remains a viable alternative path if AECIS prefers SSO
  through their M365 tenant later — same FastAPI plumbing, different
  authlib provider config.

WHY
  - Inspectors don't all have Microsoft accounts; Google works for
    everyone with a Gmail / Workspace login.
  - Per-user attribution makes the daily-quota, statistics-collection,
    and 14-day-verification features actually function.
  - Tenant separation moves from "manual env var" to "claim from
    JWT" — multi-customer use unlocks.

REDIRECT URI
  Already configured: https://hse.aecis.ca/auth/callback (Google
  Cloud Console → APIs & Services → Credentials → "violation-bot").
  JS origin remains https://hse.aecis.ca.

AZURE AD (FUTURE, NOT NOW)
  Will be added as a SECOND provider once Google OAuth is shipped
  and stable. Same FastAPI plumbing — different authlib provider
  config, different client secret. The users table schema below
  already accommodates multiple providers via (provider,
  provider_user_id) UNIQUE.

IMPLEMENTATION SKETCH
  1. Read client_id + client_secret from .env (loaded at boot from
     the staged JSON). DO NOT commit the JSON to git — it lives in
     ~/Desktop/violation-bot/secrets/, outside the repo.
  2. Use authlib (Python) — mature OAuth client lib, integrates
     cleanly with FastAPI.
  3. Routes:
       GET  /auth/login    -> redirects to Google authorize URL
       GET  /auth/callback -> exchanges code for tokens, fetches
                              user profile (email, oid, name), creates
                              or upserts a `users` row, sets a session
                              cookie (signed JWT, httponly, samesite=lax,
                              secure).
       POST /auth/logout   -> clears cookie
  4. Schema:
       users (
         id UUID PRIMARY KEY,
         provider TEXT,            -- 'google' (or 'azure_ad' if added)
         provider_user_id TEXT,    -- Google's `sub` claim, or Azure oid
         email TEXT,
         display_name TEXT,
         picture_url TEXT,         -- Google's profile picture URL
         tenant_id UUID,           -- which AECIS tenant this user belongs to
         created_at TIMESTAMPTZ,
         last_login_at TIMESTAMPTZ,
         UNIQUE(provider, provider_user_id)
       )
  5. FastAPI dependency: get_current_user(request: Request) -> User
     Reads cookie, verifies JWT, returns User row (or 401).
  6. Decorate every existing /api/* route with this dependency. Each
     route gets `user: User = Depends(get_current_user)`.
  7. Replace DEFAULT_TENANT_ID logic with `user.tenant_id` everywhere:
       - /api/upload writes photos.tenant_id = user.tenant_id
       - /api/pending filters by user.tenant_id
       - /api/batches filters by user.tenant_id
       - /api/photos/{id}/* checks photo.tenant_id == user.tenant_id
         before letting the user touch it
  8. Frontend: <header> shows logged-in user's email + sign-out link.
     Anonymous visit -> redirect to /auth/login.

DEPENDENCIES / CAVEATS
  - Multi-tenant requires the `tenants` table to be populated with
    real customer rows (not just "Public Demo"). Provisioning UI for
    AECIS admins to add new tenants is its own mini-feature.
  - The mobile-app's POST endpoint also needs auth. Either:
       (a) Same OAuth, mobile uses a different redirect URI
       (b) Service-account API keys (stored in mobile app keychain)
  - Some endpoints might need to remain public (e.g. /healthz). Keep
    them allowlisted in the dependency.
  - Use python-jose for JWT signing in our own session cookies.

EFFORT
  ~3-5 days for full multi-tenant cutover. Most of the work is the
  per-route plumbing, not the OAuth handshake itself.

3.  FREE TIER: 30 PHOTOS / DAY / USER
=============================================================

WHAT
  Per-user daily quota of 30 photo classifications. Hard cap; over
  the limit returns 429 with a "upgrade for more" message.

WHY
  - Cost containment as the user base grows (each photo is ~$0.003 on
    Flash + Stage 2, so 30/day/user = ~$2.70/user/month if they max
    out — sustainable)
  - Forces conversion to paid tier for power users
  - Discourages script-bombing

IMPLEMENTATION SKETCH
  1. Schema:
       quota_usage (
         user_id UUID,
         day DATE,
         photos_classified INT DEFAULT 0,
         PRIMARY KEY (user_id, day)
       )
  2. /api/upload increments quota_usage for the current user + UTC day.
     If user is over QUOTA_FREE_PER_DAY (default 30 from env), return
     HTTP 429 with JSON {error: "daily_limit_reached", limit: 30,
                          resets_at: "<next UTC midnight ISO>"}.
  3. Frontend: catch 429, show the existing error banner with
     i18n("quota_reached") + countdown to UTC midnight.
  4. Footer adds a small "today: N / 30 photos used" indicator next
     to the existing cost ticker.
  5. Optional: per-tier limits (free=30, paid=unlimited). Add
     `users.tier` column, gate on that.

DEPENDENCIES / CAVEATS
  - Depends on auth (each request needs to know user_id).
  - 30 is generous-but-tight. Adjust via env QUOTA_FREE_PER_DAY.
  - Counting per-photo, not per-upload-batch — a 50-photo batch from
    a free user uploads the first 30 and rejects the next 20 with a
    clear error.

EFFORT
  ~half a day after auth is in place.

4.  USER + USAGE STATISTICS
=============================================================

WHAT
  Track and surface basic usage telemetry: daily active users,
  photos uploaded per user, photos reviewed (= corrections submitted)
  per user, time-to-review, batch sizes, accuracy by inspector.

WHY
  - AECIS account managers need to see who's using the product
  - Inspector productivity dashboards
  - Identify power users for retention; identify lapsed users for
    re-engagement
  - Monitor cost-per-active-user

IMPLEMENTATION SKETCH
  1. Most signal already exists in the existing tables (corrections,
     classifications, photos). Just need a query layer + UI.
  2. New table for daily roll-ups (compute nightly via cron):
       daily_user_stats (
         day DATE,
         user_id UUID,
         photos_uploaded INT,
         photos_reviewed INT,         -- corrections rows where action!=null
         confirms INT,                -- action='confirm'
         corrections_count INT,       -- action='correct'
         active BOOL,                 -- did anything today
         PRIMARY KEY (day, user_id)
       )
  3. Roll-up job (scripts/rollup_daily_stats.py): runs from cron at
     00:30 UTC, computes yesterday's row per user, inserts.
  4. Admin-only routes:
       GET /admin/stats               -> overview (DAU, MAU, total
                                          uploads, review rate)
       GET /admin/stats/users         -> per-user table
       GET /admin/stats/users/{id}    -> per-user time series
  5. Admin gate: `users.is_admin = TRUE` flag, dependency
     `get_admin_user`.
  6. Charts: a small JS lib (chart.js or just SVG) — the data is
     small (one row per user per day, ~100 rows max).
  7. Per-batch summary already exists at /api/export/summary. Extend
     it with reviewer attribution once auth is in.

DEPENDENCIES / CAVEATS
  - Depends on auth (every action needs user_id).
  - Privacy: the stats are "internal management only" — never expose
    other users' usage to non-admin users.
  - Compliance: if you collect personal data on European users,
    consider GDPR. AECIS is Vietnam-based so probably not relevant
    initially, but plan for it.

EFFORT
  ~2 days for the roll-up + basic admin dashboard.

5.  EMAIL OUTPUT (ZIP / CSV / HTML / PDF)
=============================================================

WHAT
  Inspector clicks "Email this batch" instead of downloading; gets
  the export delivered to their email (PDF, ZIP, CSV, HTML — all
  current export formats already exist in webapp/app.py).

WHY
  - Mobile users can't easily download multi-MB ZIPs through a
    browser; email gets it onto their corporate inbox where IT
    workflows already handle large attachments
  - Senior inspectors review on phone; they want to forward straight
    to the safety officer
  - Audit trail: there's a "sent on YYYY-MM-DD to X" record that the
    raw download doesn't produce

IMPLEMENTATION SKETCH
  1. Pick an SMTP provider:
       - SendGrid (free tier 100/day) — easiest
       - AWS SES — cheap if AECIS already has AWS
       - Microsoft Graph (since they're on M365) — best integration
         with Azure AD account
       Recommendation: Gmail API via the same Google OAuth project
       that handles login, since the credential is already in. Falls
       back to SendGrid if delivery to non-Google addresses gets
       throttled.
  2. Schema:
       email_jobs (
         id UUID, user_id UUID, batch_id UUID,
         format TEXT,            -- 'pdf' | 'zip' | 'csv' | 'html'
         to_email TEXT,
         status TEXT,            -- 'pending' | 'sent' | 'failed'
         error TEXT,
         created_at TIMESTAMPTZ,
         sent_at TIMESTAMPTZ
       )
  3. New endpoint:
       POST /api/export/email
         body: {batch_id, format, to_email}
         enqueues an email_jobs row; returns 202 + job id
  4. Background worker (extend webapp.worker or a new email_worker):
       - polls email_jobs.pending
       - generates the export by calling the existing in-process
         export functions (export_pdf, export_zip, etc.)
       - sends via Graph or SES with the file as attachment
       - updates email_jobs.status
  5. Frontend: dropdown menu has "Download" (current) + "Email to me"
     button. Form for "Send to: <other email>" optional.
  6. Throttle: 5 emails per user per day. Larger ZIPs (>20MB) require
     "premium" tier or get truncated.

DEPENDENCIES / CAVEATS
  - PDF/ZIP generation already exists; this is purely the delivery
    channel.
  - Large attachments: ZIPs of 200-photo batches can be 50-200MB.
    Some email systems reject >25MB. Fallback: upload to R2 with a
    presigned URL and email the LINK instead of the file.
  - Spam risk: rate-limit per recipient address, not just per sender.
    Otherwise users could spam external addresses with auto-generated
    reports.

EFFORT
  ~3 days. The email worker + delivery layer is the new code; the
  export generation is reusing what's there.

6.  USER-SUBMITTED HSE CLASSES (ADMIN-APPROVED)
=============================================================

WHAT
  Inspectors can propose a new HSE category or fine sub-type that's
  not in the AECIS canonical taxonomy. The proposal queues for admin
  review; on approval it joins the live classifier vocabulary.

WHY
  - AECIS taxonomy isn't static — new violation types emerge with
    new project types, new equipment, new regulations
  - Without this, inspectors hack around with the "note" field;
    valuable data gets buried in unstructured text
  - Distributes taxonomy curation across the user base instead of
    requiring a central edit cycle

IMPLEMENTATION SKETCH
  1. Schema:
       hse_class_proposals (
         id UUID PRIMARY KEY,
         proposed_by UUID REFERENCES users(id),
         parent_slug TEXT,           -- which existing parent it goes under
         proposed_slug TEXT,         -- generated from label
         label_en TEXT,
         label_vn TEXT,
         description TEXT,           -- why this class is needed
         example_photo_ids UUID[],   -- inspector points at example photos
         status TEXT,                -- 'pending' | 'approved' | 'rejected'
         reviewed_by UUID,
         reviewer_note TEXT,
         created_at, reviewed_at
       )
  2. User-side flow:
       - Edit form gets a "Propose a new sub-type" link below the
         picker
       - Modal: parent (pre-selected from current pick) + label_en +
         label_vn + description + "use this photo as an example"
         checkbox
       - POST /api/hse-classes/propose
       - User can save a correction tagged with the PROPOSED slug;
         the correction goes through normally but the slug is stored
         as 'pending:<uuid>' until approved, then renamed.
  3. Admin review flow:
       - GET  /admin/proposals (pending list)
       - POST /admin/proposals/{id}/approve (slug becomes real, gets
         appended to fine_hse_types_by_parent.json + reloaded by
         workers + retroactively patched on existing corrections)
       - POST /admin/proposals/{id}/reject (with reason; user
         notified)
  4. The taxonomy file gets HOT-RELOADED on every approval (workers
     re-read fine_hse_types_by_parent.json on a SIGHUP, or via a
     /admin/reload-taxonomy endpoint).
  5. Approval policy: require 2+ inspectors to vote-up the same
     proposal before admin sees it (community filter; reduces admin
     queue). Or skip and trust the admin to reject duplicates.

DEPENDENCIES / CAVEATS
  - Depends on auth (proposed_by is a user_id).
  - Slug collisions: enforce uniqueness across all 386 fine slugs
    + all parent slugs. Auto-suffix on conflict.
  - VN/EN both required (or the model can't render the label
    correctly for the wrong-language inspector).
  - The classifier prompt has a hard-coded 29-class parent list. If
    user proposes a new PARENT (not just sub-type), the prompt must
    re-render. Not impossible — the prompt builder reads taxonomy.json
    dynamically — but worth keeping out of v1.

EFFORT
  ~5-7 days. The hot-reload semantics + retroactive slug-patch on
  approval is the trickiest part.

7.  REAL-USER VERIFICATION AFTER 14 DAYS
=============================================================

WHAT
  After a user has been active for 14 days, prompt for additional
  identity verification (SMS code, email magic-link, or similar).
  Until verified, the account is in "trial" mode (might have lower
  quota or feature gates).

WHY
  - Filters bots / abandoned-test accounts from real long-term users
  - Required before granting "real-user" privileges (higher quota,
    propose new HSE classes, email export)
  - Reduces fraud / quota-laundering across multiple free accounts

IMPLEMENTATION SKETCH
  1. Schema additions to `users`:
       phone_e164 TEXT,
       phone_verified_at TIMESTAMPTZ,
       email_verified_at TIMESTAMPTZ,
       trial_expires_at TIMESTAMPTZ,    -- created_at + 14 days
       verified BOOL DEFAULT FALSE
  2. After 14 days from first login:
       - Frontend banner: "verify your account to keep using"
       - Soft block: read-only mode (can view but not upload)
  3. Verification methods (in priority order):
       a. SMS: send 6-digit code via Twilio, user enters in modal
       b. Email magic-link: send unique URL via Graph; click verifies
       c. Domain-based: if email ends in @aecis.com or known partner
          domains, auto-verify on signup (Azure AD already validated
          the domain)
  4. Admin override: admin can manually verify a user from the
     /admin/users panel.
  5. Sub-feature: a per-IP rate limit on verification attempts (max
     5 per hour) to stop brute-force.

DEPENDENCIES / CAVEATS
  - SMS costs ~$0.01-0.05 per verification depending on provider /
    country. Vietnam SMS is cheap; western numbers are pricier.
    Budget impact: if 1000 users sign up and 80% verify by SMS,
    that's $40-80 — negligible.
  - Domain auto-verify (3c) is the cleanest UX for AECIS-internal
    users and should be the primary path. SMS is the fallback for
    external clients.
  - Storing phone numbers requires care under privacy regulations.
    Encrypt at rest + only show last 4 digits in admin UI.
  - The 14-day timer is per-user; a re-login doesn't reset it. Keep
    `trial_expires_at` immutable after first set.

EFFORT
  ~2 days for a single verification method (e.g. SMS-only). +1 day
  per additional method.

IMPLEMENTATION ORDER (updated 2026-04-29)
=============================================================

Auth is heavy and blocks the inspector-facing items. Reordering so
the user-facing improvements ship first; auth lands last as the
gating step that activates the per-user features that have been
built underneath cookie/session attribution in the meantime.

  1. PWA (#1)             FIRST — independent of auth, lifts mobile
                          UX immediately
  2. Quota (#3)           cookie-based for now, retro-fits to user_id
                          when auth ships
  3. Stats (#4)           anonymous + per-cookie / per-IP for now;
                          retro-fits when auth ships
  4. Email export (#5)    user enters destination email manually
                          (no profile lookup) until auth ships
  5. User-submitted classes (#6)  attributed to "anonymous" inspector
                          until auth ships; admin reviews regardless
  6. 14-day verification (#7)     skip until auth lands (no user
                          identity to verify)
  7. Auth (#2)            LAST — converts cookie/session attribution
                          to real user identity, unlocks #6, retroactively
                          attributes all the existing #2-5 data

Realistic phasing:

  PHASE 1 (~3 days):   [DONE] PWA — manifest, service worker, install
                       flow, offline app shell, R2 thumbnail SWR cache
  PHASE 2 (~3 days):   [DONE] Quota + stats (cookie-keyed)
                       /admin/login + /admin/stats dashboard live
  PHASE 3 (~3 days):   [DONE] Email export — "Email it" in export menu
                       falls back to 7-day R2 link for >20MB blobs
  PHASE 4 (~5 days):   [DONE] User-submitted classes — Propose-new
                       button in correct-region modal, admin queue at
                       /admin/proposals, hot-reload of taxonomy on
                       approval, retro-patch of pending: corrections,
                       one-time decision toast on user's next visit
  PHASE 5 (~3 days):   [DONE] Auth — Google OAuth + retro-attribution
                       migration of pre-auth rows. Header user-pill,
                       /auth/me JSON, /admin promotes via is_admin.
  PHASE 6 (~2 days):   14-day verification (lights up after auth)
  PHASE 7 (~1 day):    [DONE] Azure Entra ID as second OAuth provider
                       + AUTH_REQUIRED gate forcing every visitor to
                       sign in via the /auth/signin chooser before
                       reaching any other route. Whitelist: OAuth flow,
                       static assets, service worker, env-password
                       admin login.

Total: ~3 weeks of focused work.

Trade-off vs auth-first: more rework when auth lands (every per-user
feature needs a quick adapter from cookie-id -> user_id). But ships
visible value much sooner.

  appears (volume of personal-Google accounts, or quota laundering),
  light it up via:
    1. ALTER TABLE users ADD COLUMN verified BOOL DEFAULT FALSE
    2. domain-allowlist auto-verify on first sign-in (toananh.com,
       aecis.com)
    3. soft block: read-only mode for verified=false past day 14
  Estimated reactivation effort: ~½ day if domain-only, ~2 days
  with SMS fallback.

THINGS TO KEEP IN MIND ACROSS ALL OF THESE
=============================================================

  - Backwards compatibility: a chunk of existing data has no user_id
    (was inserted before auth). Phase 5 retro-attributes per-cookie
    rows on first sign-in for THAT user; rows from cookies that never
    log in stay user_id=NULL forever. Either ignore them, backfill
    with a 'legacy@' placeholder user, or surface as "anonymous" in
    admin views. Status quo (NULL) is fine — no functional impact.

  - Tenant migration: the system currently ships single-tenant
    ("Public Demo"). Multi-tenant requires a tenant-picker on signup
    and admin tooling to create new tenants. NOT in any of the 7
    items above — parallel workstream, ~5 days estimate. Triggered
    when the second AECIS client onboards.

  - Per-user cost ticker: SHIPPED as EXTRA-D. Admins still see
    global; non-admins see personal. The admin-stats panel could
    grow per-user cost columns later (currently only counts photos
    + corrections per user).

  - The audit trail (corrections.user_key, corrections.user_id) is
    already populated by Phase 5. No further migration needed.

  - Documentation: every feature above needs a short user-facing
    help doc. AECIS clients aren't going to read source code.
    Open task — write a per-feature README under docs/ when a real
    inspector hits a "how do I…?" question that's worth canonicalising.

  - Session expiry handling on the frontend: the gate redirects on
    page navigation, but a fetch() that returns 401 mid-session
    (after the 30-day session expires) is not currently caught by
    JS. Low-risk because the session is long. Future enhancement:
    intercept 401 responses and redirect to /auth/signin.

  - SW cache version: every change to the rendered shell needs a
    CACHE_VERSION bump in webapp/static/service-worker.js so PWA
    users actually pick up the new HTML. Easy to forget.

End of plan.
